Cloud tools need accounts and servers. VCS tools provide history but add workflow steps and handle binaries poorly. Syncthing/Resilio avoid servers but still have limited rollback and conflict handling. Our design targets that gap with account-free LAN sync plus LWW rollback \cite{bayou1995}. We expect shorter setup, lower LAN latency, and better recovery; these are hypotheses.

\section{Research to Application Mapping}
\begin{itemize}
	\item \textbf{Zero-configuration peer discovery on LAN}
	\begin{itemize}
		\item \textbf{Research Insight:} mDNS and DNS-SD use multicast queries and service records to find services without a central server \cite{rfc6762,rfc6763}.
		\item \textbf{Application in project}: Use Go advertise/browse on LAN so peers can join without accounts or setup.
	\end{itemize}
 
	\item \textbf{Causal ordering for concurrent updates}
\begin{itemize}
	\item \textbf{Research Insight:} Logical clocks help detect concurrent edits without a global clock \cite{lamport1978}.
	\item \textbf{Application in project:} Track per-file metadata (timestamps + vector-clock style state) before conflict handling.
\end{itemize}
 
\item \textbf{Consistency strategy under failures and partitions}
\begin{itemize}
	\item \textbf{Research Insight:} CAP shows strict consistency is hard under partitions; weakly connected systems use deterministic rules to converge \cite{brewer2002,bayou1995}.
	\item \textbf{Application in project:} Favor availability on LAN; use LWW + backups and resync after reconnection.
\end{itemize}
 
\item \textbf{Conflict-handling evolution path}
\begin{itemize}
	\item \textbf{Research Insight:} CRDTs can avoid conflicts but add state and complexity \cite{shapiro2011}.
	\item \textbf{Application in project:} Keep CRDTs for later; use LWW + backup now.
\end{itemize}
 
\item \textbf{Bandwidth-efficient synchronization direction}
\begin{itemize}
\item \textbf{Research Insight:} Delta transfer (rsync-style) reduces data sent for large or partial changes \cite{rsync1996}.
\item \textbf{Application in Project:} Start with whole-file transfer, then add block-level diffs for large files and multi-peer scaling.
\end{itemize}

\item \textbf{Local change detection primitives}
\begin{itemize}
    \item \textbf{Research Insight:} OS event APIs (inotify, FSEvents) provide change streams without polling \cite{inotify7,fsevents}.
    \item \textbf{Application in project:} Use native watchers per OS to capture add/modify/delete events and trigger sync.
\end{itemize}

\item \textbf{Content identity and transport}
\begin{itemize}
    \item \textbf{Research Insight:} SHA-256 gives content identity for integrity checks, and TCP gives reliable ordered delivery \cite{fips1804,rfc9293}.
    \item \textbf{Application in project:} Hash files for change detection and verification, and move data over TCP sockets between peers.
\end{itemize}
\end{itemize}
\section{Key References}
Key sources that ground the design choices:
\begin{itemize}
    \item \textbf{LAN service discovery:} RFC 6762/6763 define zero-configuration discovery and advertisement for local networks \cite{rfc6762,rfc6763}.
    \item \textbf{Causal ordering:} Lamport’s logical clocks establish ordering guarantees for concurrent updates \cite{lamport1978}.
    \item \textbf{Consistency trade-offs:} Brewer’s CAP perspective informs availability-first design under partitions \cite{brewer2002}.
    \item \textbf{Conflict-free replication path:} CRDTs provide a future option for conflict avoidance at the cost of added complexity \cite{shapiro2011}.
    \item \textbf{Efficient sync:} The rsync algorithm motivates later delta-transfer optimizations \cite{rsync1996}.
    \item \textbf{Implementation primitives:} SHA-256, file event APIs, and TCP support hashing, change detection, and transport \cite{fips1804,inotify7,fsevents,rfc9293}.
\end{itemize}

\section{Conclusion}
The literature suggests: make LAN discovery and sync automatic, and use simple, deterministic conflict rules.
\begin{itemize}
    \item \textbf{Key takeaways.} LAN discovery standards and ordering help collaboration, while availability-first consistency with deterministic conflict handling fits weakly connected teams.
    \item \textbf{Design influence.} We use mDNS/DNS-SD, per-file metadata, LWW + backup, and whole-file transfer as the baseline, with delta-sync and CRDTs reserved for later.
    \item \textbf{Limits in existing tools.} Cloud platforms depend on servers, VCS tools add workflow overhead, and P2P sync tools lack rollback for temporary, multi-owner teams.
\end{itemize}
Overall, we keep shared-folder UX while adding version safety. The expected advantages (setup time, LAN latency, recovery safety) remain hypotheses to test.
