\section{Problem Context}

\subsection{Scenario and Pain Points}

To motivate the problem, we begin with a concrete collaborative scenario commonly observed in university projects.\par
University students frequently work in temporary groups and need to exchange files efficiently without relying on complex infrastructure, cloud account management, or external storage devices. During the course of a project, team members collaboratively produce and modify heterogeneous files, including source code, documents, images, datasets, and presentation materials. Since collaborators often use different operating systems and development environments, cross-platform compatibility further increases the complexity of file sharing and coordination.\par
In practice, temporary student teams usually lack the time, resources, and technical expertise required to maintain a dedicated file management or synchronization system. As a result, files are often exchanged through ad hoc communication channels such as group chats or email attachments. This approach introduces several issues, including duplicated files, inconsistent versions, lack of synchronization, and absence of proper change tracking.\par
In addition, ownership and persistence of project data become problematic once the collaboration ends. Some members may wish to preserve the project files for future reference, while others may prefer to discard them. However, because the files are typically distributed informally and managed inconsistently, long-term access often depends on a single individual maintaining the archive. This creates risks of data loss, incomplete project history, and fragmented ownership.\par
Therefore, the students require a lightweight and decentralized solution that enables them to:
\begin{itemize}
    \item create a shared repository without complicated server deployment or administrative setup;
    \item support synchronization of heterogeneous file types across multiple platforms;
    \item track modifications and maintain version history;
    \item provide shared ownership of project data among collaborators; and
    \item avoid dependence on a single participant for long-term storage, archival, or deletion of the project files.
\end{itemize}

\subsection{Problem Definition}

Based on the scenario described above, small-scale collaborative groups require a file-sharing mechanism that is lightweight, decentralized, and easy to use. In such environments, participants mainly care about whether they can quickly form a shared workspace, exchange project files reliably, preserve previous versions when changes are made, and continue working with the shared folder as if it were a normal local directory.\par
From this objective, the problem can be defined as the design of a peer-to-peer directory-level synchronization system with lightweight version control. More specifically, the system must solve the following technical problems:

\begin{itemize}
\item \textbf{Local peer discovery and dynamic membership.}
The system must discover available peers on the same local network without relying on a central server. It should also support dynamic participation, where peers may join, leave, or temporarily disconnect during collaboration.

\item \textbf{Directory-level change detection and propagation.}
The system must monitor a shared local directory and detect file creation, modification, and deletion events. These changes should be propagated to other online peers without requiring users to manually upload, download, or explicitly send files.

\item \textbf{Rapid synchronization among online peers.}
When a change occurs, the system should synchronize the updated state to other available peers within a short delay, so that collaborators can observe a consistent project workspace during active collaboration.

\item \textbf{Support for heterogeneous file types and large files.}
The system must handle different categories of project files, including text files, source code, documents, images, and binary files. For large files, the system should support efficient transfer mechanisms rather than assuming that every file can be sent as a single small message.

\item \textbf{Simple versioning and rollback.}
Before a file is overwritten by a synchronized update, the system should preserve an older copy or historical state. This provides a basic recovery mechanism against accidental overwrites, incorrect updates, or unintended file changes.

\item \textbf{Transparent integration with the local file system.}
The shared workspace should behave like a normal directory on the user's computer. Users should be able to create, edit, move, and delete files using their usual tools, while synchronization and version management are handled by the system in the background.
\end{itemize}
Therefore, the central technical problem is to provide a serverless, peer-to-peer, directory-level synchronization mechanism that supports dynamic local-network collaboration, heterogeneous file transfer, rapid update propagation, and basic version recovery while preserving the simplicity of a normal local folder.

\section{Identified Gap}

University students and small collaborative teams face significant constraints when choosing file-sharing solutions for project work. The current landscape presents a fundamental gap: either users rely on centralized services with inherent limitations, or they resort to ad hoc peer-to-peer tools that lack critical versioning and conflict-resolution features.

\subsection{Limitations of Centralized Solutions}

Cloud-based file sharing services (Google Drive, GitHub, Microsoft Teams) introduce several technical and operational challenges:
\begin{itemize}
    \item \textbf{Centralized infrastructure dependency:} These services require persistent internet connectivity and rely on third-party servers, introducing latency, availability risks, and potential data privacy concerns in institutional or air-gapped environments.
    \item \textbf{Limited file type and size support:} Platforms optimized for documents or source code often struggle with heterogeneous file types (datasets, binaries, media files) and impose storage quotas.
    \item \textbf{Communication friction:} Most file sharing occurs through secondary communication channels (email, messaging apps), separating file distribution from the actual collaboration workspace.
    \item \textbf{Ownership and permission bottlenecks:} Centralized ownership models (e.g., GitHub branch protection, Google Drive permissions) introduce approval delays and reduce team autonomy. Changes may require owner confirmation (e.g., merge requests), slowing iteration cycles during active sprints.
\end{itemize}

\subsection{Limitations of Existing Peer-to-Peer Solutions}

While peer-to-peer services (AirDrop, Syncthing) avoid centralization, they present their own constraints:
\begin{itemize}
    \item \textbf{Absence of versioning:} Most P2P file-sharing tools provide no built-in history or rollback capability, making accidental overwrites irreversible.
    \item \textbf{Directory structure inconsistency:} Local P2P services often fail to maintain directory-level consistency and do not integrate seamlessly with the file system abstraction.
    \item \textbf{Limited conflict resolution:} Concurrent edits and synchronization failures lack deterministic handling strategies.
\end{itemize}

\subsection{The Gap}

The gap is clear: no lightweight, decentralized, directory-level file synchronization tool exists that combines the freedom of peer-to-peer sharing with the safety of version history, the simplicity of local network operation, and support for heterogeneous file types without permission bottlenecks.

\section{Positioning Your Project}

Our project addresses this gap by providing a lightweight, decentralized, directory-level file synchronization system designed specifically for local-network collaboration among small teams. Unlike centralized cloud services or generic P2P tools, our solution combines the best aspects of both approaches while remaining simple and portable.

\subsection{Key Differentiators}

\begin{itemize}
    \item \textbf{Local network operation:} Our system operates entirely within the local network (WLAN/Bluetooth), eliminating dependency on external servers and providing immediate, latency-free collaboration.
    \item \textbf{Heterogeneous file support:} The system places no artificial restrictions on file types or sizes. Any file readable by the operating system can be synchronized. The only practical limit is the available disk space and the OS's own capabilities.
    \item \textbf{Built-in versioning and rollback:} Unlike most P2P solutions, our system automatically preserves previous versions of files before overwriting them, providing a safety net against accidental modifications or conflicts.
    \item \textbf{Lightweight and portable:} Designed as a simple, portable application with an intuitive user interface. No server configuration, administrative privileges, or complex setup required. Users simply select a folder and begin collaborating.
    \item \textbf{Transparent file system integration:} The shared directory behaves like a normal folder on the user's computer. Synchronization, versioning, and conflict resolution happen transparently in the background, without requiring users to master specialized workflows (like Git commits or merge requests).
\end{itemize}

\subsection{Design Philosophy}

Our approach prioritizes \textbf{simplicity}, \textbf{autonomy}, and \textbf{reliability}:
\begin{itemize}
    \item \textbf{Simplicity:} Minimize user friction. File sharing and versioning should "just work" without requiring users to understand distributed systems, conflict resolution, or version control concepts.
    \item \textbf{Autonomy:} No ownership bottlenecks. Any peer can modify files; changes propagate to all peers without requiring approval or synchronization ceremonies.
    \item \textbf{Reliability:} Preserve data safety. Version history protects against accidental overwrites; deterministic conflict resolution ensures consistent behavior across all peers.
\end{itemize}

\section{Research to Application Mapping}
\begin{itemize}
	\item \textbf{Zero-configuration peer discovery on LAN}
	\begin{itemize}
		\item \textbf{Research Insight: } mDNS + DNS-SD provide lightweight discovery without a central server in local networks
		\item \textbf{Application in project}: Implement Go-based peer discovery with automatic advertise/browse behaviour; use WLAN as primary transport.
	\end{itemize}

	\item \textbf{Casual ordering for concurrent updates}
\begin{itemize}
	\item \textbf{Research Insight:} Logical clocks and event ordering reduce ambiguity when multiple peers edit concurrently.
	\item \textbf{Application in project:} Track per-file version metadata (timestamps + vector-clock style state) in the coordination layer before resolving conflicts.
\end{itemize}

\item \textbf{Consistency strategy under failures and partitions}
\begin{itemize}
	\item \textbf{Research Insight:} CAP trade-oofs show that perfect consistency is difficult during partition/offline scenarios.
	\item \textbf{Application in project:} Favor availability and eventual convergence on LAN; use deterministic conflict policy (LWW(Last-Write-Wins) + backup copy) and resync after reconnection.
\end{itemize}

\item \textbf{Conflict-handling evolution path}
\begin{itemize}
	\item \textbf{Research Insight:} CRDTs can avoid some mutual conflict handling but increase design complexity.
	\item \textbf{Application in project:} Keep CRDT as a future extension; current releases uses simpler LWW(Last-Write-Wins) + version-backup workflow for predictable behavior.
\end{itemize}

\item \textbf{Bandwidth-efficient synchronization direction}
\begin{itemize}
\item \textbf{Research Insight:} Delta-based transfer (rsync-style) reduces unnecessary data movement for large or partially changes files.
\item \textbf{Application in Project:} Start with whole-file transfer for implementation simplicity, then add block-diff optimization for larger files and multi-peer scaling. 
\end{itemize}
\end{itemize}

\section{Conclusion}
\begin{itemize}
	\item \textbf{What did you learn from the literature?}
	\begin{itemize}
		\item Practical LAN P2P sync is best grounded in mature discovery standards (mDNS/DNS)
		\item Concurrent updates require careful version tracking and deterministic conflict resolution for predictable user experience.
        \item Perfect consistency is not achievable during partitions; favor availability and eventual convergence in local
        \item CRDTs are promising for future conflict-free replication but add complexity; simpler LWW + backup is a good starting point.
        \item Delta-based synchronization can optimize bandwidth for large files but should be added after a working whole-file transfer implementation.
    \end{itemize}
    \item \textbf{How did the literature influence your design?}
    \begin{itemize}
        \item We chose mDNS/DNS-SD for peer discovery to avoid central servers and simplify LAN collaboration.
        \item We implemented per-file version metadata and a deterministic LWW conflict policy to handle concurrent edits predictably.
        \item We designed the system to favor availability and eventual convergence during partitions, with resync after reconnection.
        \item We deferred CRDT implementation to future versions, starting with a simpler versioning and backup approach for conflict handling.
        \item We prioritized whole-file transfer for initial implementation, with plans to add delta-sync optimization later.
    \end{itemize}
    \item \textbf{What are the limitations of the existing work?}
     \begin{itemize}
        \item Many existing solutions either require central servers (Google Drive, GitHub) or lack versioning and conflict resolution (AirDrop).
        \item Some systems support versioning but do not handle dynamic peer discovery or local synchronization well (Git).
        \item Few solutions provide a seamless directory-level sync experience with heterogeneous file support and simple version recovery for small-scale LAN collaboration.
    \end{itemize}
    \item \textbf{What are the next steps in your project?}
    \begin{itemize}
        \item Implement the core peer discovery and directory synchronization features based on the insights from the literature.
        \item Test the system in real-world LAN environments with multiple peers and various file types to evaluate performance and usability.
        \item Plan for future enhancements, including CRDT-based conflict resolution and delta-sync optimization, based on user feedback and observed limitations.
    \end{itemize}
\end{itemize}
